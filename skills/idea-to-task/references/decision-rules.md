# SCQA 推演与风险边界

## 1. SCQA 推演

**所有输入统一走这一节。** 不再先做战略型、归纳型或常规型分流；先在内部完成一轮有序推演，再由推演结果自然导出后续处理方式。

当你处理输入时，按下面顺序在内部完成推演：

1. **提取背景**
   - 当输入直接给出业务现状、已有动作、组织范围或时间背景时，优先采用这些明确信息。
   - 当输入没有说全，但上下文已经暗示了前提、目标对象或约束时，可以补出最小必要背景，不要为了凑完整而扩写。
2. **提取矛盾**
   - 当输入明确提到痛点、变化压力、时点要求、资源限制或推进阻力时，把这些视为打破现状的关键矛盾。
   - 当表面上只有方向口号，但语气里已经包含“必须现在做”“不能再拖”“多线协同会失控”等压力信号时，也要识别为矛盾线索。
   - 当出现跨部门协同、资源竞争、治理节奏、多业务线对齐、组织权责不清等战略级信号时，即使没有明说痛点也要提取为矛盾，避免漏判公司级推动事项。
3. **推导核心问题**
   - 核心问题必须从“背景”和“矛盾”的张力里自然长出来，回答“现在到底要解决什么”。
   - 当现有张力无法支撑一个稳定的问题时，不要硬造问题，转去第 2 节判断是否需要追问。
4. **推导核心判断**
   - 核心判断必须直接回应上一步的问题，给出可行动的方向，而不是“先分析一下”“先看看再说”这类回避表述。
  - 当多个主题本质上都在服务同一个核心判断时，直接收敛为一个顶层方向；原来的多主题只作为后续里程碑或子任务线展开，不再单独设置归纳步骤。

内部推理统一使用以下格式，且只允许存在于内部：

```xml
<scqa_analysis>
S: [从输入中提取的背景]
C: [从输入中提取的矛盾]
Q: [从张力中推导出的核心问题]
A: [对核心问题的方向判断]
完整度: [强/中/弱]
</scqa_analysis>
```

明确限制：

- 用户可见输出只呈现自然语言的“核心判断”，不出现任何方法论术语，也不暴露内部推理过程。
- 不把推演写成并行清单；必须先看背景，再看矛盾，再推问题，最后落到方向判断。

术语泄漏负面示例（以下表达绝对不能出现在用户可见输出中）：

- 错误：「根据 SCQA 分析，当前 Situation 是…，Complication 在于…」→ 正确：直接给结论「当前重心应放在统一治理节奏上」
- 错误：「经过 Question 推导，核心 Answer 是先做平台化」→ 正确：「核心判断：优先平台化」
- 错误：「基于金字塔原理的顶层结论…」「按 MECE 原则拆分为…」→ 正确：直接写结论和拆分，不提方法论
- 错误：在输出中保留 `<scqa_analysis>`、`<thinking>` 或任何内部标签 → 正确：只输出六段式正文

### 特殊输入类型处理

当推演结果显示这是一个跨多条业务线、需要统一目标和治理节奏、且周期跨季度的公司级推动事项时，核心判断必须保留“先做总体方案设计”的约束：

- 第一主题先落在总体方案，而不是直接跳到工具、部门动作或预算项。
- 总体方案至少覆盖目标与边界、现状与约束、治理机制、推进路线。
- 其余主题再按“可并行的诊断线”和“必须后置的落地线”展开。

识别规则在此节为单一事实源；模板侧（output-template.md 第 3 节）按本节触发条件填充战略型条件分支，不复写触发规则。

### 混合信号优先级

当战略型与归纳型信号同时命中时（例如「公司级治理节奏 + 单顶层意图」这类同时具备公司级变革信号与归纳收敛潜力的输入），优先按战略型组织：第一主题仍为「总体方案设计」，归纳收敛的统一目标融入总体方案的「目标与边界」里程碑或作为推进路线图的顶层表述，不单独压成「单主题（归纳）」形态。

三类触发的优先级矩阵为：**战略型 > 归纳型 > 常规型**。与此矩阵一致的判定结果写入 `<scqa_analysis>` 的 `完整度` 同级或作为内部标注，不暴露给用户。

## 2. SCQA 追问触发

只有当关键要素缺失并且会改变任务方向时，才触发定向追问。SCQA 定向追问与后续通用补问合并计数，总问题数不超过 5 个；当两类问题同时触发时，优先把 SCQA 问题合并到补问列表中，避免重复或冗余。

当出现以下情况时触发追问：

- **只有模糊背景，没有关键矛盾**：例如只有愿景口号或泛泛方向，无法判断当前挑战是什么。此时追问应聚焦“现在卡在哪里”或“为什么此刻要做”。
- **背景和矛盾都在，但可能导向两条相反方向**：例如同样的现状既可能导向“先平台化”，也可能导向“先单点验证”。此时追问应聚焦决策倾向，而不是补背景细节。
- **输入过短，四项信息都提不起来**：例如只有一句口号、一个名词或一个模糊目标。此时追问应补背景、约束和成功标准。

当出现以下情况时不追问：

- 背景和矛盾已经足以推出稳定方向，即使把握不是百分之百。
- 输入不完美，但主线已经清楚，只差执行层细节。

追问表达要求：

- 每个问题都要说明“这个答案会如何改变任务方向”。
- 问题必须短、具体、能直接影响主题划分、优先级或推进路径。

## 3. 证据强度与 SCQA 完整度

证据强度由内部推演完整度决定，不再依赖旧的前置分流。

- **强证据**：四项信息齐全，背景、矛盾、核心问题、核心判断都能稳定成立，且方向可以直接落到行动层。
- **中等证据**：背景和矛盾明确，核心问题与核心判断需要结合上下文推演，但推演路径基本唯一。
- **弱证据**：只有模糊背景，关键矛盾不清楚，或同一输入还能合理导出多条相反方向。

证据强度与分层任务树的深度按以下弹性规则联动（详见 [output-template.md](./output-template.md) 第 3 节层级规则）：

- 弱证据：展开到主题层，即 2 层结构（核心判断 → 主题）
- 中等证据：展开到里程碑层，即 3 层结构（核心判断 → 主题 → 里程碑）
- 强证据：展开到执行项层，即 4 层结构（核心判断 → 主题 → 里程碑 → 执行项）

上述层级允许 ±1 层弹性；当输入天然只能支撑 2 层时（极弱证据），不强造第三层，停在主题层并在「风险与高风险推断」中标注不确定性。

## 4. 高风险歧义

出现以下场景时，优先双路径输出：

- “先做高端还是先做大众”
- “先做平台还是先做单点功能”
- “现在投入还是继续观察”

## 5. 常见基线偏差

没有专用 skill 时，模型常出现这些偏差：

- 只做摘要，不做任务树
- 会提一些问题，但不给推进顺序和角色建议
- 看到模糊内容就平均分配篇幅，没把主线压出来
- 能看出两条路线，但不会显式标记高风险歧义

你的目标是修正这些偏差，而不是把内容写得更长。

## 6. MECE 互斥完备校验

**触发时机**：金字塔输出生成前，在 SCQA 推演完成之后、任务树渲染之前执行一次。对用户透明，推理过程放入 `<mece_check>` 标签，不进入最终输出。

**四维校验**（QUAL-01 全覆盖）：

1. **同层互斥**：同一层的支撑点语义必须彼此排斥，不得出现“提升效率 vs 优化流程”这类表述重叠的并列项。发现重叠时，合并同类项或改用更具区分度的措辞。
2. **同层完备**：同一父节点下的支撑点必须覆盖父节点所要回答的全部关键维度；若覆盖不足，在「风险与高风险推断」段落显式标注缺口（例如「当前拆分未覆盖 X 维度，存在推进盲区」）。
3. **数量 2-4**：每一层的支撑点数量必须落在 2 到 4 之间（与 output-template.md §3、SKILL.md 工作流步骤 5 字面一致）；低于 2 或高于 4 触发第 7 条 underflow/overflow 规则。
4. **分组一致性**（QUAL-03 落地）：每层同层分支必须采用同一分组标准。`<mece_check>` 内对每层显式标记 `grouping: inductive | deductive`：
   - `grouping: inductive` — 分支是从多条观察事实归纳上升而来，校验时检查“分类标准是否一致”（如全部按业务线拆 / 全部按阶段拆，不得混用）。
   - `grouping: deductive` — 分支是从上层结论演绎下降而来，校验时检查“因果推导链是否完整”（子节点是否确实是父节点的必要支撑，不得跳链）。
   - 同层分支不得混用两种 grouping。

**内部推理格式**：

```xml
<mece_check>
层级: [主题层 | 里程碑层 | 执行项层]
分支: [分支1, 分支2, ...]
grouping: [inductive | deductive]
互斥性: [通过 | 发现重叠：...]
完备性: [通过 | 缺口：...]
数量: [N 个，状态：通过 / underflow / overflow]
分组一致性: [通过 | 不一致：...]
处置: [通过直出 | 合并 | 补充 | 触发 §6 cardinality 规则]
</mece_check>
```

**弱证据退化**（D-03）：

当 §3 判定为弱证据且输出只到主题层（2 层结构）时，`<mece_check>` 自动退化为仅执行「数量 2-4」单维检查；不再要求互斥性 / 完备性 / 分组一致性维度，避免对无分层的输出过度校验。

**输出隔离纪律**：

- `<mece_check>` 标签及其内部字段（`grouping`、`inductive`、`deductive`、`互斥性`、`完备性` 等）绝对不得出现在用户可见输出中。
- 用户只看到结构正确的金字塔任务树，看不到校验过程。
- 术语泄漏禁令详见 SKILL.md「明确禁止」段。

**Cardinality 兜底规则**：

当「数量 2-4」维度检查命中 underflow 或 overflow 时，按以下规则处置，`<mece_check>` 不越权接管补问或重构金字塔。

- **Underflow（同层 <2 个稳定候选）**：
  1. 视为「输入过窄」，优先由 §2 SCQA 追问机制在上游捕获；若追问已跳过或无法追加，则进入兜底。
  2. 兜底退化为 2 层结构（核心判断 → 里程碑），即取消主题层，直接以里程碑承接核心判断。
  3. 在「风险与高风险推断」段落显式标注：「输入不足以支撑多主题拆分，已退化为单主题结构」。
  4. `<mece_check>` 仅输出 `处置: underflow 兜底 → 2 层结构`，不尝试拼凑第 2 个主题。

- **Overflow（同层 >4 个候选）**：
  1. 优先触发归纳聚类，复用 §1 第 4 步已定义的归纳收敛能力，尝试将相近候选合并为 2-4 个主题。
  2. 若无法自然合并（候选语义互斥、共同上位概念不成立），选取与核心判断距离最近的 top 3 作为主题，其余候选降级处理：
     - 符合里程碑粒度的，归入某个保留主题下作为里程碑。
     - 仍保持独立方向但证据不足的，写入「需补问」清单或「风险与高风险推断」段落。
  3. `<mece_check>` 输出 `处置: overflow 聚类 → N 个主题 + M 个降级项` 或 `处置: overflow top3 + 补问清单`。

- **纪律约束**：
  - underflow 情况下 MECE 不得越权触发追问；追问权归 §2，§6 只做结构兜底。
  - overflow 聚类必须可逆向回溯（原候选在 `<mece_check>` 内留痕），不允许静默删除候选。
