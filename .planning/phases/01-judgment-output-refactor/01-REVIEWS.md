# Cross-AI Plan Review — Phase 1

## Gemini Review

通过对 Phase 1 的两个执行计划（01-01 和 01-02）进行深度评审，该方案准确捕捉到了 skill 输出"平铺化"的痛点，并提出了针对性的"向上归纳"重构路径。

### 1. 总体评估 (Summary)
该计划设计严密，通过在 `decision-rules.md` 中引入"发散再收敛"的思维模型（先拆分主题再进行归纳检查），从逻辑底层解决了父子意图被错误平铺的问题。方案巧妙地将"战略型输入"归纳为通用的"向上归纳"规则的一个特例，既保持了现有逻辑的兼容性，又扩展了 skill 处理复杂、模糊意图的能力。新增的"归纳型"输出模板填补了平铺任务与战略蓝图之间的结构空白。

### 2. 方案优点 (Strengths)
- **思维逻辑闭环**：将归纳步骤置于"主题密度判断"之后、输出之前，符合先分析细节再提取顶层意图的自然认知过程。
- **静默回退设计**：坚持"不暴露内部推理"原则，在归纳失败时选择静默走平铺逻辑，保证了输出的专业感和一致性。
- **结构化标识**：引入 `单主题（归纳）` 标签，能有效引导模型区分"天然单主题"与"收敛单主题"，对后续的任务树展开深度有显著的暗示作用。
- **平滑兼容**：将战略型规则泛化，避免了为不同场景堆砌补丁，使得 `SKILL.md` 的核心原则更加简洁且具有指导意义。

### 3. 风险与担忧 (Concerns)
- **模板选择冲突 (Medium)**：当一个输入同时符合"战略型"特征和"归纳型"特征时，模型可能会在两个模板间摇摆。建议在 `output-template.md` 中增加优先级说明。
- **节编号依赖风险 (Low)**：修改了 `decision-rules.md` 的节编号并同步更新了 `SKILL.md`，需确保没有其他外部脚本或 Agent 配置文件硬编码了原有的节标题索引。
- **里程碑降级深度 (Low)**：将原并列主题降级为里程碑后，如果原主题本身很复杂，可能会导致任务树在单一里程碑下显得过于臃肿。

### 4. 改进建议 (Suggestions)
- 明确模板优先级：若触发"战略型"判断，则强制执行"战略型"模板；其他多主题归纳成功的场景才执行"归纳型"模板。
- 增强判定标准描述：增加"去中心化"测试。
- 预热 Phase 2 示例：可以预先放入一个极其简短的对比小例子。

### 5. 风险等级评估 (Risk Assessment)
**风险等级：LOW (低)** — 纯 Prompt 工程优化，不涉及底层代码或外部依赖改动。

## Codex Review

### 总体判断

两个计划的拆分方式基本合理，依赖顺序也对：先改规则和模板，再改入口和做校验。但它们当前更像"文档结构改写计划"，还不够像"输出行为优化计划"。最大问题有两个：一是 Plan 01 明确要求"保持现有战略型模板完全不变"，这和当前最典型的失败样例存在直接冲突；二是两个计划的验证都主要停留在 grep 和结构校验，无法证明 skill 真会把"并列主题"收敛成"单顶层意图 + 子任务线"。

### Plan 01 Review

**Strengths:**
- 把"向上归纳"插在主题拆分之后、输出之前，位置设计合理
- 明确了成功、失败两种路径，并保留"静默回退"
- 输出模板覆盖了"核心判断 / 主题拆分 / 任务树"三段，改动面完整
- 保持目录边界清晰，只动目标目录内文件

**Concerns:**
- [HIGH] 计划要求"现有战略型模板完全不变"，但当前战略型模板本身就是多顶层展开结构。如果典型问题就是"全公司 AI 转型"被拆成 6 个并列主题，那不改这一段，主问题大概率还在。
- [HIGH] 缺少行为级验证。现有验证只检查文本是否出现、章节编号是否递增，并不能证明模型实际会把多主题收敛成一个顶层意图。
- [MEDIUM] 新增"多不相关主题也先尝试归纳"与现有"多不相关主题：禁止强行汇总成一个总项目"存在潜在张力。
- [MEDIUM] "归纳型核心判断"新增了句式要求，但没有显式说明它与现有"用 3 到 6 句话回答"如何兼容。

**Suggestions:**
- 明确"战略型"和"归纳型"的优先级关系
- 加一个行为验证任务：至少用 2 个 fixture 验证"平铺前/归纳后"差异
- 改写主题密度判断第 2 节的语气，把"禁止强行汇总"改成"若归纳失败则禁止强行汇总"
- 在模板中补一句优先级说明：归纳型核心判断仍需控制在 3 到 6 句内

**Risk: HIGH** — 较高概率"改了文档但没修到核心问题"

### Plan 02 Review

**Strengths:**
- 依赖关系明确，先做 Plan 01 再接入入口文件，顺序正确
- 工作流新增步骤的位置准确
- 核心原则从"战略型专用"扩展到"多主题通用"，方向是对的
- 使用 `uv run ... --strict` 做结构校验，符合仓库约束

**Concerns:**
- [HIGH] 计划把 `--strict` 验证当成 phase 完成度的重要证明，但脚本实际只校验结构，不会检查输出质量
- [MEDIUM] `must_haves` 里要求工作流新步骤和 `output-template.md` 形成明确联动，但计划中具体改法不完全覆盖
- [MEDIUM] 第 6 步保留"战略型输入第一主题固定为总体方案设计"，第 7 步说"存在目标从属关系则收敛为单顶层主题"，二者先后语义不明确

**Risk: MEDIUM** — 主要风险是"验证通过了，但目标没被验证到"

## Consensus Summary

### Agreed Strengths
- 归纳步骤位置设计合理（主题拆分之后、输出之前），两位评审一致认可
- 静默回退策略得当，符合"不暴露内部推理"原则
- 将战略型规则泛化为通用原则的方向正确
- 改动范围控制好，不越界

### Agreed Concerns
- **[HIGH] 战略型模板与归纳型模板的优先级关系不明确** — 两位评审都指出当输入同时符合战略型和归纳型特征时，模型可能会在两套模板间摇摆。需要明确优先级。
- **[HIGH] 缺少行为级验证** — Codex 强调只有 grep 和结构校验不足以证明 skill 真的会把并列主题收敛为单顶层意图。Gemini 也建议增加对比案例。
- **[MEDIUM] "禁止强行汇总"与"先尝试归纳"存在规则张力** — 两者措辞可能给模型发出矛盾信号。

### Divergent Views
- **风险评估分歧**：Gemini 评 LOW（纯 Prompt 工程），Codex 评 Plan 01 HIGH（认为可能"改了文档但没修到核心问题"）。分歧核心在于是否认为"战略型模板不变"会导致典型失败样例仍然失败。
- **战略型模板处理**：Gemini 建议保留战略型模板并加优先级说明；Codex 质疑"保持战略型模板完全不变"本身就是问题，因为典型失败场景恰好走的是战略型路径。
