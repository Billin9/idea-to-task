# Cross-AI Plan Review — Phase 03

## Gemini Review

这份计划旨在通过 SCQA（情景、冲突、问题、回答）方法论重构 `idea-to-task` skill 的输入解析层，是 v2.0 咨询方法论重构的核心第一步。以下是详细评审：

### 1. 总体评价 (Summary)
该计划设计精良且极具战略性，通过「先推演后分流」的逻辑，成功地将 v1.0 碎片化的判断规则整合进了统一的咨询方法论框架中。它不仅完美遵循了 `PROJECT.md` 的约束（如隐藏术语、不改目录结构），还针对 `PITFALLS.md` 中提到的「指令过载」和「方法论冲突」做了针对性防御。工作流的彻底重构为 Phase 4 的金字塔输出打下了坚实的逻辑基础。

### 2. 设计优点 (Strengths)
- **统一逻辑框架：** 废除了 v1.0 僵硬的前置分流（战略型/归纳型/常规型），改由 SCQA 推演结果自然驱动处理方式，极大地提升了系统对复杂、模糊输入的鲁棒性。
- **XML 推理隔离：** 使用 `<scqa_analysis>` 标签进行内部推理，既利用了 LLM 的思维链能力，又通过「显式禁止」确保了方法论术语不会污染用户侧的输出。
- **证据强度联动：** 建立了 SCQA 完整度与证据强度的直接映射，将「信息质量」转化为了「任务树深度」的控制变量，逻辑闭环完整。
- **风险前置防御：** 针对 `PITFALLS.md` 中的「指令过载」风险，将 8 节规则精简为 5 节，并采用有序步骤而非规则平铺，显著降低了模型执行偏差。
- **向上归纳的自然融合：** 将 v1.0 复杂的「向上归纳检查」合并到 Answer 推导中，简化了逻辑复杂度。

### 3. 潜在问题与风险 (Concerns)
- **LOW:** 「战略型」特征识别的模糊性。战略型判断不再前置后，如果输入非常碎片，AI 可能在推演阶段漏掉跨部门协同的信号。
- **LOW:** 追问计数的复杂性。SCQA 定向追问与通用补问分开计数，理论上可能导致问题总数过多。
- **MEDIUM:** Answer 映射的稳定性。`output-template.md` 尚未在 Phase 3 修改，如果 SCQA 推导出的 Answer 长度或语气与现有模板的“核心判断”不匹配，可能导致输出风格突变。

### 4. 改进建议 (Suggestions)
- 在 `decision-rules.md` 第一节的“提取矛盾”规则中，增加对“跨部门”“资源竞争”“治理节奏”等战略信号的显式引导。
- 在 `SKILL.md` 中补充约束：当触发 SCQA 定向追问时，应尽量和通用补问合并，总问题数控制在 5 个以内。
- 增加一个关于“术语泄漏”的负面示例，明确哪些表达绝对不能出现在用户输出中。

### 5. 风险评估 (Risk Assessment)
**风险等级：LOW**

理由：计划高度对齐研究结论，采用“推倒重来”而非“增量补丁”的策略，有效避免了新旧逻辑冲突。任务定义清晰，验证手段可执行。

**审批意见：PASS（准予执行）**

## Consensus Summary

这轮 cross-AI review 的总体判断是：**Phase 03 的计划方向正确，方法论重构思路成立，风险可控，但在执行前应收紧少数边界契约并控制指令密度。**

### Agreed Strengths
- 两个 reviewer 都认可 Phase 03 的核心方向是正确的：用 SCQA 单入口替代旧的前置分流，能显著提升对 CEO 碎片输入的解析一致性。
- 都认可 `<scqa_analysis>` 作为内部推理隔离层是好设计，既利用了模型内部推演能力，又能降低术语泄漏风险。
- 都认为计划与 `PROJECT.md`、`PITFALLS.md` 的约束对齐较好，没有明显的越界或结构性误判。

### Agreed Concerns
- **弱证据边界契约需要严格一致。** the agent 直接点出“弱证据边界不一致”，这后来也确实在执行后的 verifier 中被验证为真实 gap。
  追踪说明：该问题已在执行阶段通过提交 `c713535` 修复，不再是当前代码状态的阻塞项。
- **Task 1 的规则密度偏高。** 尽管方案方向正确，但把太多规则压进一个大任务，容易让执行时出现遗漏或局部漂移。
- **术语与输出接口的验收标准需要更细。** 尤其是“核心判断”的语气、长度和术语禁止项，最好始终保持可 grep / 可 spot-check 的字面标准。

### Divergent Views
- Gemini 的结论更积极，给出 `LOW` 风险和可立即执行的判断。
- the agent 更谨慎，把整体风险看作 `LOW-MEDIUM`，并把 3 个中等级问题提前上升为执行前应关注的项。
- Gemini 更关注 Phase 3 与 Phase 4 的接口稳定性；the agent 更关注 Phase 3 内部规则文本的一致性和可执行性。

### Takeaway
- 如果把这份 review 用于后续 phase 规划，最值得继承的不是“重做 SCQA”，而是三条规划纪律：
  1. 规则边界必须在 `PLAN.md`、`SKILL.md`、`decision-rules.md` 三处保持完全一致。
  2. 大任务里涉及多个概念迁移时，要优先把可验证的边界写成字面标准。
  3. 与下游 phase 的接口要提前声明，但不要在当前 phase 提前吞并后续范围。
