# Architecture Patterns: 咨询方法论集成到 AI Skill Prompt

**Domain:** Markdown-based AI skill prompt engineering (consulting methodology integration)
**Researched:** 2026-04-14
**Overall confidence:** HIGH (基于实际文件分析 + 方法论标准文献)

## 推荐架构：四层处理管线嵌入现有文件结构

### 核心设计决策

**不新增文件，不改目录结构。** 四个方法论层（SCQA、Pyramid、MECE、So-what）嵌入现有四个文件中，而非为每个方法论单独建 reference 文件。

理由：
1. PROJECT.md 明确约束「只修改 skills/idea-to-task/ 目录下的四个文件，不改目录结构」
2. 当前三个 reference 文件的职责划分（判断规则 / 输出模板 / 示例）已经覆盖了方法论需要落地的位置
3. LLM 的 reference 加载是按需触发的——拆成更多小文件会增加加载判断复杂度，对 token 效率无正向帮助

## 各层详细设计

### Stage 1: SCQA 推演（核心新增）

**职责：** 将 CEO 碎片输入结构化为四要素，作为后续金字塔构建的输入。

**与现有逻辑的关系：**
- **替换**原步骤 2（战略型前置判断）、步骤 3（主题密度判断）、步骤 4（可任务化程度判断）
- 原来的「战略型判断」变成 SCQA 推演的一个输出属性（当 Complication 涉及多部门变革 + 统一治理时自动标记）
- 原来的「向上归纳检查」被 SCQA 的 Answer 层自然吸收——Answer 本身就是归纳后的顶层结论

**SCQA 四要素在 skill 中的语义：**

| 要素 | 咨询含义 | 在 idea-to-task 中的语义 | 来源 |
|------|---------|------------------------|------|
| S (Situation) | 背景共识 | CEO 的业务现状和已有条件 | 从输入中提取 |
| C (Complication) | 打破现状的变化/挑战 | CEO 面临的核心问题或变化 | 从输入中提取 |
| Q (Question) | 核心需要回答的问题 | CEO 真正需要解决的是什么 | 推演得出 |
| A (Answer) | 顶层结论 | 输出任务树的顶层意图 | 推演得出，即金字塔顶点 |

**推演 vs 追问的判断：**
- AI 先基于输入尝试补全 S/C/Q/A 全部四个要素
- 只有当 C（变化/挑战）或 Q（核心问题）完全无法从输入推演时，才触发追问
- S（背景）缺失通常可以合理假设，不构成追问理由
- A（答案）由 AI 基于 S+C+Q 推演，不需要追问

**落地位置：**
- SKILL.md：工作流步骤 2 改为「SCQA 推演」，简述流程
- decision-rules.md：新增「SCQA 推演规则」章节，定义四要素提取方法、追问触发条件、战略型输入的 SCQA 特征

### Stage 2: 金字塔构建（核心重构）

**职责：** 从 SCQA 的 Answer（顶层结论）出发，向下分层展开任务树。

**弹性 3x3 结构：**
```
Level 0: Answer（顶层结论/意图）        ← SCQA 的 A
    │
    ├── Level 1: 战略支柱 1             ← 最多 3 个
    │       ├── Level 2: 关键任务 1.1   ← 每个支柱下最多 3 个
    │       ├── Level 2: 关键任务 1.2
    │       └── Level 2: 关键任务 1.3
    │
    ├── Level 1: 战略支柱 2
    │       └── ...
    │
    └── Level 1: 战略支柱 3
            └── ...
```

**弹性规则：**
- 证据强度决定展开深度（与现有 decision-rules.md 第 6 节「证据强度与下钻深度」对齐）
- 证据弱：只到 Level 0（项目级）
- 证据中等：到 Level 1（里程碑级/战略支柱级）
- 证据强：到 Level 2（执行项级/关键任务级）
- 每层分支数量是上限 3，不是必须 3——MECE 校验会确保不遗漏，但不硬凑

**与现有模板的关系：**
- **替换**现有的「战略型/归纳型/常规型」三套模板
- 金字塔结构天然统一了这三种场景：战略型只是 Level 1 支柱更多且总体方案为首、归纳型是多输入收敛到单 Answer、常规型是直接展开
- 原来的三路判断（战略型→归纳型→常规型）被简化为单一金字塔构建流程

**落地位置：**
- SKILL.md：工作流步骤 3-4 改为「金字塔构建」
- output-template.md：用金字塔模板替换现有三套模板，保留六段输出框架（核心判断/主题拆分/任务树/执行建议/风险/下一步）

### Stage 3: MECE 校验（内部逻辑）

**职责：** 对金字塔每层的节点做不重叠、不遗漏检查。

**校验规则：**
- **不重叠（ME）：** 同层两个节点的职责/范围不能有交集。如果有，合并或重新划分
- **不遗漏（CE）：** 同层节点合起来是否覆盖了上层节点的全部范围。如果有明显缺口，补充

**实施方式：**
- 纯内部逻辑，不在输出中暴露
- 在金字塔构建完成后、格式化输出前执行
- 如果校验发现问题，AI 自行调整节点，不追问用户

**落地位置：**
- SKILL.md：工作流步骤 5 简述「内部执行 MECE 校验」
- decision-rules.md：新增「MECE 校验规则」章节，定义重叠判断标准和遗漏检查方法

### Stage 4: So-what 过滤（内部逻辑）

**职责：** 对每个任务节点执行「所以呢？」测试，剔除无行动意义的空洞描述。

**过滤规则：**
- 每个节点必须能回答「这个任务完成后，会推动什么具体行动或产出什么具体交付物？」
- 如果答案是「不知道」或「只是一个方向声明」，降级为上层节点的描述文字，而非独立任务
- 特别针对：「加强XX能力」「提升XX水平」「完善XX体系」这类空洞表述

**实施方式：**
- 纯内部逻辑，不在输出中暴露
- 在 MECE 校验之后执行
- 过滤掉的内容不是删除，而是吸收到上层节点的描述中

**落地位置：**
- SKILL.md：工作流步骤 6 简述「内部执行 So-what 过滤」
- decision-rules.md：新增「So-what 过滤规则」章节，定义空洞描述的判断标准和降级策略

## 与现有逻辑的兼容性分析

### 保留不变的部分

| 现有能力 | 为什么保留 | 在新架构中的位置 |
|---------|-----------|----------------|
| 输入类型识别（长文本/碎片/混合） | 仍然需要，是管线的第一步 | Stage 0，SKILL.md 步骤 1 |
| 证据强度与下钻深度 | 控制金字塔展开深度 | 融入 Stage 2 的弹性规则 |
| 高风险歧义双路径输出 | 核心能力，SCQA 的 C 层自然承载 | Stage 1 的 Complication 判断 |
| 补问数量控制（2-5 个） | 用户体验约束 | Stage 1 的追问规则 |
| 六段输出框架 | 输出稳定性，用户已习惯 | Stage 5，保持不变 |
| 明确禁止清单 | 防御性规则 | SKILL.md 保留 |

### 被替换的部分

| 现有逻辑 | 替换为 | 理由 |
|---------|-------|------|
| 战略型前置判断（decision-rules 第 1 节） | SCQA 推演中的战略型特征识别 | SCQA 的 S+C 自然包含战略型判断所需的信息 |
| 主题密度判断（decision-rules 第 2 节） | 金字塔 Level 1 分支数量 | 金字塔构建天然决定了主题数量 |
| 向上归纳检查（decision-rules 第 3 节） | SCQA 的 Answer 推演 | Answer 就是归纳后的顶层结论，不再需要单独的归纳步骤 |
| 三套模板（战略型/归纳型/常规型） | 统一金字塔模板 + 弹性深度 | 金字塔结构统一了三种场景 |
| 可任务化程度判断（decision-rules 第 4 节） | SCQA 推演的完整度 | S/C/Q/A 四要素的完整度就是可任务化程度的衡量 |

### 需要重写的示例

| 现有案例 | 处理方式 | 理由 |
|---------|---------|------|
| 案例 1：输入清楚 | 重写，展示完整 SCQA→金字塔流程 | 需要展示新管线的标准路径 |
| 案例 2：输入混杂 | 重写，展示 SCQA 如何处理多主题 | 原来的「向上归纳」被 SCQA Answer 替代 |
| 案例 3：高风险歧义 | 重写，展示 SCQA 的 Complication 如何承载歧义 | 框架变了 |
| 案例 4：证据不足 | 重写，展示金字塔弹性深度 | 需要与新的深度控制机制对齐 |
| 案例 5：战略型正例 | 重写，展示 SCQA 如何自然识别战略型 | 不再有单独的战略型前置判断 |
| 案例 6：非战略型反例 | 重写或保留精简 | 需要验证新管线不会把简单项目过度结构化 |
| 案例 7：归纳型正反对比 | 重写，用金字塔 vs 平铺对比 | 归纳机制变了 |

## 建议构建顺序（依赖驱动）

### Phase 1: SCQA 输入解析层

**改动文件：** decision-rules.md（新增 SCQA 章节）→ SKILL.md（重构工作流步骤 2）

**为什么先做：**
- SCQA 是管线入口，所有后续阶段依赖 SCQA 的输出
- SCQA 的 Answer 是金字塔的顶点——没有 Answer 就无法构建金字塔
- SCQA 推演规则独立性强，不依赖后续阶段

**交付物：**
- decision-rules.md 新增「SCQA 推演规则」章节
- SKILL.md 工作流步骤重构（原步骤 2-4 合并为 SCQA 推演）
- 1-2 个 fixture 验证 SCQA 推演行为

### Phase 2: 金字塔原理输出结构

**改动文件：** output-template.md（重构模板）→ SKILL.md（工作流步骤 3-4）

**为什么第二：**
- 依赖 Phase 1 的 SCQA Answer 作为金字塔顶点
- 是用户可感知的最大变化（输出格式变了）
- 需要与 SCQA 配合测试

**交付物：**
- output-template.md 用金字塔模板替换三套模板
- SKILL.md 工作流步骤更新
- 1-2 个 fixture 验证金字塔输出

### Phase 3: MECE + So-what 内部校验

**改动文件：** decision-rules.md（新增两个章节）→ SKILL.md（工作流步骤 5-6）

**为什么最后：**
- MECE 和 So-what 是金字塔的质量校验层，必须在金字塔构建完成后才有意义
- 两者都是内部逻辑，对用户不可见，可以一起做
- 依赖 Phase 2 的金字塔结构作为校验对象

**交付物：**
- decision-rules.md 新增「MECE 校验规则」和「So-what 过滤规则」两个章节
- SKILL.md 工作流步骤更新
- 复用 Phase 1-2 的 fixture，增加 MECE/So-what 期望行为

### Phase 4: 示例校准与兼容性验证

**改动文件：** examples.md（全面重写）→ 现有 fixture 更新

**为什么最后：**
- 示例必须展示完整管线行为，需要前三个 Phase 全部完成
- 正反对比案例需要基于最终确定的管线设计
- 是最终的行为锚定，任何先写的示例可能因后续阶段调整而失效

**交付物：**
- examples.md 全面重写（预计 5-7 个案例）
- 更新 tests/fixtures/ 中的 fixture 文件
- 端到端验证所有路径

## 反模式（明确不做）

### 反模式 1: 每个方法论一个 reference 文件
**做法：** 新建 references/scqa-rules.md、references/mece-rules.md、references/so-what-rules.md
**为什么不做：** 违反项目约束（不改目录结构）；增加 LLM 的加载判断复杂度；现有文件的职责划分已经能承载

### 反模式 2: 在输出中暴露 SCQA 框架
**做法：** 输出中显示「S: ...  C: ... Q: ... A: ...」
**为什么不做：** PROJECT.md 明确要求 MECE/So-what 不暴露；SCQA 是 AI 的思维工具不是用户的阅读格式；核心判断部分已经承载了 Answer 的语义

### 反模式 3: 保留三套模板 + 叠加金字塔
**做法：** 在战略型/归纳型/常规型模板上各自叠加金字塔逻辑
**为什么不做：** 金字塔结构天然统一了三种场景；保留三套会导致规则膨胀和冲突；v2.0 的核心价值就是用统一框架替代分叉逻辑

### 反模式 4: 先写示例再定规则
**做法：** 先在 examples.md 写好案例，再反推规则
**为什么不做：** 示例是规则的具象化，规则未定时写的示例会反复返工；正确顺序是规则→模板→示例

## 可扩展性考虑

| 场景 | 当前设计 | 未来扩展点 |
|------|---------|-----------|
| 新增方法论层（如议题树） | 管线可插入新 Stage | 在 MECE 和 So-what 之间插入 |
| 多语言输出 | 当前只支持中文 | output-template.md 可增加语言选项 |
| 与其他 skill 复用 | 方法论规则写在 decision-rules.md 中 | 可抽取为共享 reference（但当前不需要） |
| 更复杂的输入（带数据/表格） | SCQA 的 Situation 层可承载 | 不改架构，只扩展 S 的提取规则 |
